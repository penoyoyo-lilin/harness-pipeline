⏸ 人类审批 #1：需求确认

人类操作流水线行为
确认 / 通过更新 approval_status → approved，继续进入架构设计
提出修改意见重新调用 /analyze 传入修改意见，再次提交审批
驳回更新 status → failed，终止流污线

执行步骤

  1. 理解用户需求（模糊则主动追问关键信息）
  2. 分析需求，拆解功能模块，识别依赖关系
  3. 基于模板生成需求文档（6 大章节）
  4. 定义 API 定约若案和数据模型若案
  5. 定义错误码（5 位数字：前 3 位 HTTP 状态码 + 后 2 位细分）
  6. 更新任务状态文件和模块索引
  7. ⏸ 暂停，等待人类确认需求

⏸ 人类审批 #2：架构审批

人类操作流污线行为
确认 / 通过更新 approval_status → approved，继续进入 UI 设计
提出修改意见重新调用 /architect 修订方案，再次提交审批
驳回更新 status → failed，终止流污线

执行步骤

  1. 验证前置条件（需求 approval_status = approved）
  2. 读取需求文档和全局架构文档
  3. 读取编码规范，确保设计合规
  4. 设计 Go Domain 分层结构（types → config → repository → service → handler → router）
  5. 细化数据库模型（GORM 标签、迁移策略、索引设计）
  6. 细化 API 定约（RESTful 风格、DTO 定义、统一响应格式）
  7. 产出可直接复制编译的 Go 代码骨架
  8. 基于模板生成技术方案文档
  9. ⏸ 暂停，等待人类审批架构

📤 输出

输出路径格式说旻
Domain 分层代码internal/domain/<module>/Gotypes → config → repository → service → handler → router
单元测试internal/domain/<module>/**/*_test.goGo表驾动测试（gomock/testify mock）

实现顺序

  1. types（数据结构定义，零外部依赖）
  2. config（配置结构体与加载逻辑）
  3. repository（接口定义 + 实现，数据库操作）
  4. service（业务逻辑，可单元测试）
  5. handler（HTTP 通配层，请求/响应映射）
  6. router（路由注册，中间件挂载）
  7. 每层编译通过后再进下一层

📊 覆盖率目标

层级最低覆盖率
Service 层 / 关键组件≥ 80%
Handler 层 / API 路由≥ 70%
Repository 层≥ 60%

执行步骤

  1. 识别本次变更涉及的模块和文件
  2. 为每个模块生成表驾动单元测试
  3. 执行测试并收集覆盖率
  4. 失败修复：最多自动重试 3 次
  5. 超过重试次数则标记需要人工干预并报告
  6. 生成覆盖率报告和测试报告

🔍 审查 5 大维度

维度检查内容严重程度
🔷 架构合规分层依赖方向、模块边界🔴 严重
🔷 代码质量命名、复杂度、DRY🟡 重要
🔷 错误处理Go 不吞错、前端有养底🔴 严重
🔷 安全性SQL 注入、XSS、认证🔴 严重
🔷 性能N+1 查询、大事务、重演染🟡 重要

⚖ 审查结譾判定

结譾条件后续动作
✔ 通过无严重问题，重要问题 ≤ 2 个继续创建 PR
⚠ 需要修改1 个严重 或 3+ 个重要
❌ 拒绝架构违规 / 安全问题退回编码 Agent，可能需要重新设计
